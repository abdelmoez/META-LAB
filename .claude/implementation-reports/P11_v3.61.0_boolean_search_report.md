# P11 — Guided Boolean Search Strategy Studio (generator ↔ critic loop) — Implementation Report

- **Phase:** P11 (70.md)
- **Version:** v3.61.0 (continuing the repo's 3.x semver per 70.md's escape clause; the prompt's P11→v0.11.0 maps here to v3.61.0).
- **Feature flag:** `searchStrategyStudio` — **OFF by default**, and gated to require the whole trio ON (`searchStrategyStudio && searchEngine && pecanSearch`) since it functionally builds on the Strategy Builder (concepts) and the Automated Run (real connectors). With any of the three off, the wizard renders exactly as before.
- **User-facing name:** "Strategy Studio" / "Search Optimizer" / "Search Critic". No user-facing "AI" wording anywhere.

## What was implemented

A generator→critic→refine loop that turns PICO concepts into database-specific Boolean strategies, **tests them with real connector hit counts** (PubMed esearch `rettype=count`, OpenAlex `meta.count` — reusing the existing Pecan Search connectors, not reinventing), iterates to improve them, stores every iteration with its critique, produces **PRISMA-S search documentation**, and estimates recall against a new per-project **seed-study** set. Builds on P1 connectors + the 60/69.md Strategy Builder & versioning; adds the previously-missing seed/recall layer.

## Files changed

## Security / privacy

- All model/provider calls are server-side; the generator/critic/recall math is pure and deterministic. Real external calls (hit counts) go only through the existing rate-limited/throttled/backed-off Pecan connectors.
- The optimize loop is **bounded** (max iterations × databases, databases capped at 7) and wall-clock-guarded — no runaway external fan-out. Every iteration stores provenance (string, live count, critic JSON, change notes, timestamp).
- Reads gate on project view, mutations on edit (leader/owner) — server-side, not frontend-only. Only legal count/metadata queries; no paywall interaction.

## Performance

- Reuses the connectors' preview cache + per-provider throttle + HTTP backoff. `optimize` is synchronous but iteration- and time-bounded so it can't hang a request. No server-wide lag.

## Known limitations & concrete solutions

1. **`optimize` is synchronous (bounded ≤25s), not a durable job.** For very slow providers a request could approach the guard.
   *Solution:* promote `optimize` to the existing durable `pecanSearchWorker` job pattern (claim/drain/kick/recover) with UI progress polling — the iteration store already models per-step state, so this is a mechanical lift; documented as the next step.
2. **Real hit counts wired for PubMed + OpenAlex only** (the two the connectors expose counts for).
   *Solution:* the renderer registry + `countFetcher` are provider-agnostic; adding Embase/Scopus/WoS/Cochrane counts is a connector + one `registerRenderer` call each.
3. **Recall `probe` mode issues one membership query per seed.** Large seed sets = many small external calls. — **RESOLVED in the recs round:** `probe` is now hard-capped at `PROBE_MAX_SEEDS=25` per request (bounds external calls); the response returns `probeCapped`/`probedCount`/`probeMax` so the UI can tell the user to use `run` mode for the full set. `run` mode (single intersection) remains preferred for large seed sets.
4. **Recall title-fallback uses a similarity threshold.** Seeds with no DOI/PMID and a very different title phrasing can be missed.
   *Solution:* resolve seeds through CrossRef/OpenAlex on entry (P15 adds exactly this resolver) to attach identifiers before matching.

## Recommended next improvements

- Make `optimize` a durable job (limitation 1).
- Feed seed resolution from P15's CrossRef/PubMed/OpenAlex resolver.
- Surface the critic's before/after count deltas as a small sparkline in the timeline.
