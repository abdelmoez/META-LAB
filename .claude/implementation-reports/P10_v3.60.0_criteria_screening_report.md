## What was implemented

A zero-training, deterministic, criteria-based eligibility screening workflow that complements the existing trained relevance model during the cold-start phase (before enough human labels exist). Reviewers define structured yes/no inclusion/exclusion questions; the engine evaluates each record's title/abstract/full-text against every criterion and returns, per criterion, a `yes | no | unclear` answer with a confidence, a short rationale, a **verbatim quoted evidence sentence**, and the source field it came from. These roll up to a suggested include/exclude/unclear decision. Reviewers adjudicate; an optional, governed auto-apply can act only under strict gates and **never overwrites a human decision**. A validation view compares suggestions to human decisions.

## Security / privacy considerations

- All evaluation is server-side and **deterministic** (no browser→model calls; no external provider by default — the engine is lexical/concept-coverage over the record text the project already holds).
- Governed auto-apply is gated by: global `enabled` && `!killSwitch` && per-project `enabled` && `policy==='auto'` && confidence ≥ threshold && **no existing human decision**. Auto-applied decisions are written under a dedicated non-human reviewer id (`eligibility-engine`) so they can never share a row with — thus never overwrite — a human decision; every one is audited and reversible (owner/leader `undo`).
- Full provenance stored per assessment: engineVersion, configVersion, criteriaVersion, timestamp, confidence, quoted evidence, reviewer action. Criteria edits are audited (before/after + author).
- Permission checks are server-side (owner/leader for criteria/settings/evaluate/validation; screening perm for adjudication) — not frontend-only.

## Performance considerations

- Bulk evaluation runs on the durable `ScreenEligibilityJob` queue off the request thread (atomic claim, attempts cap, heartbeat, crash recovery), so large projects don't lag the server. Small scopes (≤25 records) run inline for snappy feedback; larger scopes return `202 {jobId}` with UI progress polling.
- Engine is pure/allocation-lean and reuses the already-tuned concept extractor; no O(N²) cross-record work.

## Known limitations & concrete solutions

1. **Lexical (concept-coverage) evaluation, not semantic.** Confidence comes from honest term/concept match strength, so paraphrased eligibility signals can read as `unclear`.
   *Solution:* the engine already exposes an `eligibility` hybrid signal and the config carries an engine-version registry — add an optional server-side semantic scorer (reuse the existing `aiEmbeddingClient` server-proxy + `EmbeddingCacheEntry`) as a second, flag-gated component fused via the existing renormalizing hybrid, keeping the deterministic path as the default/fallback.
2. **Full-text is used only when already attached to the record.** Evaluation reads `fullText` if present, else title/abstract.
   *Solution:* wire P9's full-text retrieval (`FullTextRetrievalJob`) as an upstream step so criteria that need full text can request it before evaluation; the `sourceField` already distinguishes `fullText` vs `abstract` so recall-by-source is measurable.
3. **Parallel job worker instead of folding into `ScreenAiJob`.** A separate `ScreenEligibilityJob` table/worker was used (the AI job files were outside the server agent's ownership boundary), so there are now two near-identical durable workers.
   *Solution:* a small follow-up can unify them behind one `kind`-dispatched worker; behavior is identical today, this is purely DRY consolidation.
4. **Per-criterion validation needs adjudicated data to be meaningful.** With few human decisions the per-criterion agreement table is sparse. — **RESOLVED in the recs round:** the validation panel now shows an explicit "Preliminary — based on only N decided records; record ~M more" banner below `PRELIM_MIN=30`, so small samples are never presented as a settled baseline (metrics were already NaN-safe).
