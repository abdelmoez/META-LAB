# 72.md — User Tier Management Infrastructure — Implementation Report

- **Version:** v3.67.0 (continues the repo's 3.x semver).
- **Extends** the 67.md tier backbone (ProductTier + entitlementService + Ops→Tiers) — does not duplicate it. Gated by the existing `tierSettings.enforcementEnabled` kill-switch; new admin surfaces are admin-only + additive. No user-facing "AI".

## What was implemented

The subscription-ready foundation on top of 67's tiers/entitlements: full **assignment history/audit**, an admin **tier dashboard + analytics**, a richer **per-user tier editor** (change-type + effective dates + expiration + revert), **subscription placeholder** structure, explicit **backfill**, and precedence wiring so a tier's record cap actually governs imports.

## Files changed

**Database (`server/prisma/schema.prisma`, additive; postgres mirror synced — drift test green)**
- `UserTierAssignment` — the history/audit table: userId, tierId, previousTierId, assignedById/Name, changeType (manual|promotion|downgrade|trial_start|trial_end|beta_access|institution|payment|support_override|correction|backfill|other), reason, notes, effectiveFrom, effectiveUntil, isCurrent, reverted, revertedAt/By, metaJson. Indexed by user/tier/current.
- `TierSubscription` — billing PLACEHOLDER (no payment processing): status (none|trialing|active|past_due|canceled), provider + provider customer/subscription/price/plan ids, current period, trial window, cancelAtPeriodEnd, lastPaymentAt, nextRenewalAt, failedPaymentCount. `@@unique([userId])`.

**Server**
- `services/entitlementService.js` — `recordTierAssignment` (history writer, flips prior `isCurrent`), `revertTierAssignment`, analytics helpers, `backfillUserTiers()` (idempotent — assigns default/internal tier + an initial `backfill` history row for users with no history), and `planRecordLimitFor` (owner tier's `screening.maxRecordsPerProject`).
- `controllers/tierAdminController.js` — `updateUserTierAdmin` rewritten to write a history row (change-type/effective-until/notes); new `getTierAnalytics`, `getUserTierHistory`, `getUsersInTier`, `exportUsersInTier` (CSV), `revertUserTier`, `getUserSubscription`, `updateUserSubscription`.
- `routes/admin.js` — the new admin routes wired (all `requireAdmin`).
- `src/shared/entitlements.js` — added the 72-requested entitlement keys (exports/search/screening runs-per-month, storage, collaborators, advanced synthesis, guided screening, citation mining, dashboard sharing, priority queue, support level) to `ENTITLEMENT_KEYS` + `DEFAULT_TIERS` (existing keys/values unchanged; `entitlements.test.js` updated in lockstep).
- **Precedence wiring** — `screening/uploadLimit.js` `planLimit` is now fed from the owner's tier at the two `resolveScreeningUploadLimit` call sites (`screeningController.js`, `citationMiningService.js` via `planRecordLimitFor`), so a tier record cap governs imports (UNLIMITED → no cap).
- `index.js` — `backfillUserTiers()` boot hook (fire-and-forget, after `seedProductTiers`).

**Admin UI (`AdminConsole.jsx` + `adminApiClient.js`)**
- Tier analytics dashboard (total users, per-tier count + %, avg days in tier, unassigned; recent changes/promotions/downgrades, trial, expiring-soon lists).
- Users-in-tier table (paginated/searchable, per-row Change-tier/History/Subscription, CSV export).
- Richer per-user tier editor modal (new tier + change-type + required reason + effective-until + notes).
- Per-user tier-history timeline modal with a Revert action on the current entry.
- Subscription placeholder panel ("future billing — no payments processed").
- Section id `tiers` / `nav-tiers` testid unchanged; existing enforcement + tier cards + assign panel intact.

## Role vs Tier vs Entitlement vs Subscription

Kept strictly separate (as 67 established): **role** (user/mod/admin — access; admin/mod bypass tiers), **tier** (`User.tierId` + `UserTierAssignment` history — level of access), **entitlements** (per-tier feature/limit map resolved by `resolveUserEntitlements`), **subscription** (`TierSubscription` — billing lifecycle placeholder). No field mixes them.

## Backfill behavior

`backfillUserTiers()` runs once at boot: users with no assignment history get their default tier (admins/mods an internal tier if defined) + one initial `backfill` history row; guarded on history existence so restarts are no-ops. 67's read-time default (`resolveUserEntitlements` null→default) remains as a safety net.

## Known limitations & concrete solutions

1. **Live integration re-verification pending** — the CI unit gate (incl. postgres drift) + both agents' own test runs are green, but the `api-tier-management` integration suite should be re-run against a freshly-restarted server on this build to confirm end-to-end (the dev server was mid-cycle at hand-off). *Solution:* restart `npm run dev` and run the integration suite (recs round).
2. **New entitlement keys are data-model + resolver ready but not all enforced** (exports/search/screening runs-per-month, storage, collaborators, etc.). *Solution:* add `requireEntitlement`/`requireLimit` at the relevant call sites incrementally (the keys + tier values already exist).
3. **Subscription is a placeholder** — no Stripe/payment processing. *Solution:* a future billing integration writes the `TierSubscription` fields from provider webhooks; the tier system already reads tier separately from subscription so it plugs in without a rewrite.

## Deployment notes

Additive `db push` (2 new tables + entitlement JSON defaults); the enforcement kill-switch + admin-only surfaces mean zero behavior change for end users. Backfill is idempotent and non-blocking at boot.
