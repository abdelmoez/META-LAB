# 71.md — Search Engine UX/Workflow Redesign — Implementation Report

- **Version:** v3.66.0 (continues the repo's 3.x semver).
- **Feature flag:** `searchWorkspaceV2` — **OFF by default** (requires `searchEngine`). When OFF the existing 3-step `SearchWizard` renders **byte-identical**; when ON the redesigned staged `SearchWorkspace` renders. Safe, reversible redesign.
- **No user-facing "AI" wording** added (guarded by a test across all 8 stages).

## What was implemented

A calmer, guided, progressive-disclosure **staged Search Workspace** that keeps ALL functional power. It is a **composition** redesign: it reuses every existing engine component (Search Builder, the quality/versions/export/strategy-studio/recall panels, the Pecan run surface + duplicate review) and re-arranges them into one coherent 8-stage flow — it does **not** rewrite the search engines or reimplement any feature. This fixes the current "stepper inside a phase inside a stepper" doubling, the overloaded Build screen, dense chips/tables, buried actions, and three competing headers.

## Stage → composition (all features preserved)

| Stage | Reused component(s) → feature |
|---|---|
| Research Question | PICO/question summary (`pico` prop) |
| Concepts | `SearchBuilderTab phase="define"` — concept extraction, keyword/synonym |
| Terms & Vocabulary | builder define phase — MeSH suggestions (kept separate from free-text) |

All 16 preserved-feature checklist items are reachable. The heavy Search Builder is mounted once and kept mounted (hidden off-builder-stages) so concepts/strategy survive stage switches (its autosave to the `search` WorkflowModuleState is the source of truth). Advanced Boolean/raw/logs stay in each component's existing `Disclosure`/`<details>` expanders.

## Preserved features & UX improvements (before → after)

- **Before:** 3-step wizard wrapping the builder's own 5 steps (doubled model); Build screen stacked DBs + per-DB query + estimates + quality + versions + studio; dense multi-affordance chips + 8-column run tables; buried "ready for screening"; `window.prompt`/`confirm` for versions; three headers.
- **After:** one 8-stage rail; one focused surface per stage; advanced detail behind expanders; Send-to-Screening promoted to a first-class stage; a single unified header. (The version panel's `window.prompt` and the dense chip/table internals live inside the reused components and were left intact to avoid forking engine internals — noted below.)

## Known limitations & concrete solutions

1. **`readyForScreening` two-writer race** — both `SearchWorkspace` (Send stage) and `SearchBuilderTab` own this advisory flag; editing a concept after marking ready can let the builder's next autosave rewrite it from its mount-time value. The primary handoff (post-run "Go to Screening" deep link) is unaffected. *Solution (recs round / follow-up):* lift `readyForScreening` to a single owner (a dedicated field write or a small builder prop) so there's one writer.
2. **Dense chip/table internals + `window.prompt` in the versions panel live inside the reused components** — not changed here (modifying them was out of the composition scope to avoid regressions to the flag-OFF path). *Solution:* a follow-up can calm those internals behind the same flag once the composition is validated.

## Deployment notes

Additive; flag OFF so the search experience is unchanged until an admin enables `searchWorkspaceV2` (which also requires `searchEngine`). No server behavior change (pure composition over existing APIs).
