# Complete Rebuild of the Pecan Extraction Engine

You are responsible for rebuilding the data-extraction experience in PecanRev from the ground up.

This is not a cosmetic redesign, a small refactor, or a patch to the current extraction interface. Treat it as a complete product-level rethinking of how systematic-review researchers extract data from full-text articles.

The new system must be called:

# Pecan Extraction Engine

The objective is to create a state-of-the-art extraction engine that is powerful enough for advanced systematic-review researchers while remaining calm, intuitive, and easy to understand for first-time users.

The current extraction implementation is not satisfactory and should not constrain the new design. However, do not delete existing functionality, components, services, database models, or extraction logic merely because the interface is being rebuilt. Inspect everything currently available, identify useful functionality, and reuse, refactor, or adapt it when appropriate.

The final result must feel like a purpose-built research workspace rather than a collection of forms.

---

# 1. Core Mission

Build the best possible human-in-the-loop data-extraction engine for systematic reviews and meta-analyses.

The engine must support three complementary extraction methods:

1. Intelligent table extraction
2. Click-to-capture extraction from the PDF
3. Manual data entry

The system must not depend on unreliable full automation.

AI and document intelligence should assist the researcher, but the researcher must remain in control of every extracted value.

The experience should make extraction:

* Faster
* More accurate
* Easier to verify
* Easier to audit
* Less repetitive
* Less visually overwhelming
* More reliable for downstream statistical analysis
* Suitable for large and complex systematic reviews

You have permission to improve, expand, and refine the concept beyond the instructions below when doing so creates a better product. Do not limit yourself to the exact wording of this prompt. Think like a product architect, systematic-review methodologist, senior UX designer, document-intelligence engineer, data engineer, and QA lead.

However, do not remove or weaken any requirement explicitly marked as mandatory.

# 9. Extraction Workflow Overview

The extraction workflow should support three methods in this order:

1. Table Extractor
2. Click-to-Capture
3. Manual Entry

This order reflects the preferred progression from the fastest assisted method to direct manual entry.

The user should be free to switch between methods at any time.

A single study may use all three methods.

For example:

* Baseline characteristics may come from a table.
* A sample size may be captured by clicking the text.
* A clarification note may be entered manually.

The extraction record must preserve the source and method used for each value.

# 10. Method One: Intelligent Table Extractor

The Table Extractor should be the first and most prominent extraction option.

Its purpose is not merely to copy a table into a spreadsheet. It must help users understand and map table content into structured extraction fields.

## Basic workflow

1. The user activates Table Extractor mode.
2. The user selects a table or table region in the PDF.
3. The system captures the selected region and associated PDF coordinates.
4. The engine detects the table structure.
5. The engine identifies:

   * Table title
   * Headers
   * Subheaders
   * Row labels
   * Column labels
   * Merged cells
   * Footnotes
   * Units
   * Intervention arms
   * Comparator arms
   * Time points
   * Outcomes
   * Sample sizes
   * Statistical measures
6. The extracted table is shown in an editable preview.
7. The engine proposes mappings between table cells and extraction fields.
8. The user reviews and confirms the mappings.
9. Confirmed values populate the extraction form.
10. Every populated value retains a link to its source table, page, and coordinates.

## Required table-extraction capabilities

The system should handle, as reliably as practical:

* Native text PDFs
* Scanned PDFs
* Tables with visible borders
* Borderless tables
* Multi-row headers
* Multi-column headers
* Merged cells
* Superscripts
* Subscripts
* Footnote symbols
* Confidence intervals
* Mean and standard deviation formats
* Median and interquartile range formats
* Percentages
* Fractions
* Event counts
* Sample sizes
* Units
* P-values
* Adjusted and unadjusted results
* Multiple intervention groups
* Multiple comparator groups
* Multiple time points
* Repeated outcomes
* Tables spanning multiple pages
* Continuation tables
* Tables containing both text and numeric data
* Negative numbers
* Decimal values
* Scientific notation
* Ranges
* Missing-value symbols
* “Not reported” indicators
* Values embedded in parentheses
* Cells containing more than one statistic

## Table selection

Provide an intuitive way to select a table.

Possible interaction:

* Drag a rectangle around a table.
* Automatically detect likely table boundaries.
* Allow the user to adjust the detected boundary.
* Allow selection across multiple pages.
* Show a magnified preview during selection.
* Support cancellation and retry.

## Table preview

Before mapping, show a structured preview of the recognized table.

The user must be able to:

* Edit incorrectly recognized text
* Split cells
* Merge cells
* Delete irrelevant rows or columns
* Mark headers
* Mark footnotes
* Correct decimal values
* Correct OCR mistakes
* Rename arms
* Identify time points
* Exclude irrelevant sections

* Compare the structured preview with the original PDF region

## Intelligent field mapping

The engine should use:

* Existing project extraction schemas
* Field names
* Field descriptions
* Outcome definitions
* Intervention definitions
* Comparator definitions
* PICO information
* Previously mapped tables
* Current article context
* Units
* Statistical patterns
* Header relationships
* Study arms
* Time points

to propose mappings.

Each proposed mapping should explain:

* Which table cell is being used
* Which extraction field it will populate
* Why the mapping is proposed
* Any transformation being applied
* The confidence of the proposal
* Any ambiguity detected

Never silently populate ambiguous data without showing the user.

## Bulk mapping

Allow users to map an entire row, column, arm, outcome, or time point efficiently.

Examples:

* Map an entire baseline-characteristics row to study-arm fields.
* Map intervention and control columns simultaneously.
* Map all follow-up time points for one outcome.
* Map event count and total sample size together.
* Map mean, standard deviation, and sample size as a set.

## Table memory

Where safe and appropriate, learn from confirmed mappings within the project.

For example:

* If “Tx” was confirmed as the intervention arm, remember it for that article.
* If a table format repeats across publications from the same trial, suggest the previous mapping.
* If the same extraction schema is used repeatedly, prioritize commonly selected fields.

This learning must be transparent and reversible.

Do not allow cross-project data leakage.

# 11. Method Two: Click-to-Capture

When table extraction is unsuitable, the user should be able to activate Click-to-Capture mode.

This method should make extraction from narrative text, figures, tables, and isolated values extremely fast.

## Basic workflow

1. The user activates Click-to-Capture.
2. The cursor changes to indicate capture mode.
3. The user clicks or selects a value in the PDF.
4. The engine captures the selected text or recognized value.
5. The captured value appears in a temporary capture tray or floating preview.
6. The user clicks the target extraction field.
7. The value is inserted automatically.
8. The field stores a link to the exact PDF location.

The workflow should also support selecting the target field first and then clicking the PDF value.

The engine may provide both interaction styles and remember the user’s preference.

## Capture types

Support capturing:

* Single numbers
* Decimal values
* Percentages
* Sample sizes
* Means
* Standard deviations
* Standard errors
* Medians
* Ranges
* Interquartile ranges
* Event counts
* Confidence intervals
* P-values
* Dates
* Durations
* Dosages
* Units
* Text labels
* Intervention names
* Outcome names
* Participant characteristics
* Quotations or supporting text
* Multi-part values

## Smart parsing

When a user captures text such as:

* “24.6 ± 3.2”
* “18/120”
* “15.0%”
* “HR 0.74, 95% CI 0.61–0.90”
* “Median 12 months, IQR 8–18”
* “n = 214”

the engine should parse the components and offer appropriate destination fields.

For example:

* Mean: 24.6
* Standard deviation: 3.2

or:

* Effect estimate: 0.74
* Lower confidence interval: 0.61
* Upper confidence interval: 0.90
* Effect measure: Hazard ratio

The user must confirm any multi-field parsing before final insertion.

## OCR-assisted capture

For scanned PDFs or image-based regions:

* Use OCR for the selected region.
* Show the recognized value.
* Allow immediate correction.
* Keep the source image coordinates.
* Indicate that OCR was used.
* Assign lower confidence when recognition is uncertain.

## Visual feedback

After capture:

* Briefly highlight the source location.
* Highlight the destination field.
* Show the captured value.
* Provide undo.
* Allow correction.
* Allow remapping.
* Allow users to reopen the source later.

The interaction must feel immediate and precise.

# 17. Validation System

Build a robust validation layer.

Validation should occur:

* During entry
* During table mapping
* During click-to-capture
* During save
* Before article completion
* Before analysis synchronization
* Before export

Validation types should include:

## Field validation

* Required fields
* Numeric ranges
* Decimal precision
* Invalid characters
* Invalid percentages
* Negative values where not allowed
* Missing units
* Invalid dates
* Inconsistent formats

## Cross-field validation

Examples:

* Events cannot exceed total sample size.
* Subgroup sample sizes should not exceed arm sample size.
* Percentages should approximately match event counts and totals.
* Confidence-interval lower bounds should not exceed upper bounds.
* Standard deviation should not be negative.
* End dates should not precede start dates.
* Arm totals should correspond to reported study totals where expected.
* Follow-up duration should have a unit.
* Effect measure should match the expected analysis type.
* Outcome time point should be defined.
* Mean and median should not be accidentally entered into the same role.

## Cross-arm validation

* Intervention and comparator arms must be distinguishable.
* Shared comparator handling must be explicit.
* Duplicate arm names should be flagged.
* Total randomized participants should be checked against arm totals.
* Lost-to-follow-up values should be coherent.

## Analysis-readiness validation

* Required effect inputs are present.
* Outcomes are linked to the correct arm.
* Time points are compatible.
* Units are compatible.
* Effect measures are identifiable.
* The record is not duplicated.
* The analysis population is defined when necessary.

Validation should distinguish among:

* Informational suggestions
* Warnings
* Blocking errors

Do not block users unnecessarily, but do not allow clearly invalid data to silently enter analysis.

Requirements:

* Project-based access controls
* Role-based permissions
* Secure PDF access
* No cross-project document access
* No cross-user extraction leakage
* Secure temporary files
* Safe document-processing pipelines
* Input sanitization
* Output encoding
* Protected APIs
* Audit logging
* Appropriate rate limits
* Safe file-type handling
* Validation of uploaded documents
* Prevention of path traversal
* Prevention of unauthorized direct-object access
* Careful treatment of AI-provider requests

Do not transmit PDFs or extracted content to an external AI provider unless the current architecture and product policy explicitly permit it.

When external services are used, isolate the integration and make its behavior transparent.

# 29. Responsive Behavior

The primary target may be desktop and large laptop screens because PDF extraction is inherently workspace-intensive.

However:

* The interface should degrade gracefully on smaller screens.
* Tablet layouts should remain usable.
* Small screens should not produce broken overlapping panels.
* Users should be able to switch between PDF and extraction views when side-by-side layout is not practical.
* Critical actions must remain accessible at common browser zoom levels.

# 32. Avoid False Automation

The engine must not pretend that document extraction is more certain than it is.

Never:

* Invent missing numbers
* Infer unreported values without disclosure
* Treat OCR output as unquestionably correct
* Map ambiguous cells silently
* Replace user-confirmed values without permission
* Present AI confidence as statistical certainty
* Hide transformations
* Hide assumptions
* Mark articles complete automatically without project rules
* Send invalid values to analysis silently

The engine should assist the researcher, not impersonate the researcher.

# 36. Acceptance Criteria

The implementation is not complete unless all of the following are true:

## Architecture

* The extraction backend has clear modular boundaries.
* The frontend is fully integrated into PecanRev.
* Existing project and role permissions are respected.
* Existing extraction data is preserved.
* Analysis integration is not implemented through fragile hidden coupling.

## Article workflow

* Users can view all extraction-stage articles in a usable list.
* Users can search, filter, and sort the list.
* Users can open an article in a full-screen workspace.
* Users can navigate between articles efficiently.

## PDF workspace

* The PDF loads reliably.
* Page navigation works.
* Search works.
* Zoom works without breaking the layout.
* The toolbar remains stable.
* Users can jump from a field to its source evidence.

## Extraction methods

* Table Extractor works through selection, recognition, correction, mapping, and confirmation.
* Click-to-Capture can send selected values to fields.
* Manual entry is fully functional.
* Users can switch methods at any time.
* Extraction provenance is retained.

## Data integrity

* Values are autosaved.
* Save status is honest.
* Refresh does not lose acknowledged changes.
* Validation catches common errors.
* Audit history is retained.
* Existing records remain accessible.

## Analysis integration

* Valid extracted data reaches the analysis system automatically.
* Relationships among studies, arms, outcomes, and time points are preserved.
* Synchronization errors are visible.
* Updated extraction values are reflected safely.
* Users retain control over analysis inclusion.

## Quality

* The interface is not overcrowded.
* The workflow is understandable without extensive training.
* The engine performs well on realistic projects.
* Failure states are handled gracefully.
* Accessibility requirements are met.
* Automated tests cover core workflows.
* There are no placeholder buttons or disconnected interfaces.
* There are no major console errors.
* There are no known data-loss paths.

# 38. Final Instruction

Do not approach this as a request to rearrange the current extraction page.

Approach it as an opportunity to build a flagship PecanRev engine.

Think deeply about how researchers actually work with PDFs, tables, outcomes, arms, time points, and statistical values. Reduce repetitive work without sacrificing verification. Make provenance effortless. Make errors difficult to introduce and easy to detect. Make downstream analysis reliable.

Use the current implementation as a source of functionality and context, not as a design limitation.

You have freedom to innovate, reorganize, and improve the specification when you identify a superior solution. Explain major architectural decisions through implementation documentation, but do not stop at proposing them.

Inspect first. Plan carefully. Implement completely. Test aggressively. Preserve existing data. Integrate the experience fully into PecanRev.

The final result should be a production-ready, state-of-the-art Pecan Extraction Engine that researchers would trust for serious systematic reviews and meta-analyses.
