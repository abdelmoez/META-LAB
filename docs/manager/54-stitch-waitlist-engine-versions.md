## 1. Architecture findings

### Existing waitlist architecture (preserved)
Strictly-isolated subsystem (prompt48). Separate SQLite DB via
`BETA_WAITLIST_DATABASE_URL`, its own generated Prisma client
(`server/prisma/generated/waitlist-client`), no FK to the main DB. Shared validator
`src/shared/betaWaitlist.js` is the authoritative check (client mirrors it). Public
`POST /api/waitlist` + `/api/waitlist/resend` (rate-limited, honeypot, case-insensitive
dedupe). Admin `GET/PATCH/DELETE /api/admin/beta-waitlist/*` (admin-only). Feature flag
`betaWaitlist` gates whether unauth visitors to `/` see the waitlist vs the landing
page. Confirmation email via `renderBetaWaitlistConfirmationEmail` (honest: not an
account, not an invite, no guarantee). **All of this is retained unchanged.**

### Existing Stitch design-system architecture (reused)
`src/frontend/stitch/` â€” tokens (`stitchTokens.js`, the `S.*` object â†’ `--stitch-*`
CSS vars, deep-purple brand `#5d509b`, Manrope, white cards on `#f7f9ff`), 30+
primitives (`core.jsx`, `controls.jsx`, `overlay.jsx`), shell + navConfig. CSS is
scoped under `html[data-ui-design="stitch"]`. Stitch mode is normally **admin-gated**
(`designMode.js`), but the waitlist is **public**, so the page **self-scopes**
(sets the root attribute on mount, injects `<StitchStyle/>`, wraps in `.stitch-scope`,
restores on unmount) â€” Stitch-native without depending on the admin design switch.
The latest Stitch source is the **"PecanRev Research OS"** project (Stitch MCP); the
"Beta Waitlist Page" screen informed layout + copy. We **did not** copy generated

### Discovered engine inventory
11 independently-maintained engines (see Â§3 + `docs/engine-versions.md`). Justified
exclusions: PDF viewer, PRISMA, GRADE (folded into RoB/import-export), collaboration/
notifications/presence, auth, projects CRUD, settings, citation/institution helpers
(shared infra/helpers), and the waitlist (no analytical logic, isolated DB).

### Version-storage decision
**Hybrid**: engine **catalog** (ids/names/ownership) in code
(`src/research-engine/engine-registry/engines.js`, code-reviewed, merge-safe); **live
versions + history** in the **main DB** (`EngineRegistry`, `EngineVersionHistory`,
`ProcessedEngineChange`) for atomic increments, auditability, and â€” crucially â€” because
DB-backed bumps write no files, **no recursive-CI-commit** problem. Rationale in
`docs/engine-versions.md`.

### Automation strategy
A pure, unit-tested classification core + a `scripts/engine-version.mjs` CLI
(`check`/`bump`/manual override). Decision hierarchy: manual flags > `engine-changes.json`
manifest > commit footers > rule-based-from-ownership (defaults to **minor** + warning;
reports ambiguity, fails in `--strict`). Idempotent via `(commit SHA, engineId)`.

## 4. Version-classification rules (deterministic)

`bumpVersion`: minor â†’ `{major, minor+1}`; major â†’ `{major+1, 0}` (e.g. v0.9â†’v0.10,
v1.7â†’v2.0). Classification precedence: **manual flags > manifest > commit footers >
rule-based**. Rule-based maps changed files to engines via ownership globs and ALWAYS
defaults to **minor** (never auto-major) with a warning. Shared-infra globs (e.g.
`server/index.js`, `src/frontend/components/**`, `schema.prisma`) and no-bump globs
(docs, tests, CI/build config, the waitlist + landing page, the engine-version system
itself) prevent spurious bumps. Conservative fallback: engine known but major/minor
uncertain â†’ minor + warning; engine unknown â†’ report ambiguity (fail in `--strict`).
Idempotent via `(changeKey, engineId)`; concurrency-safe via that unique constraint +
a single transaction.
