# META·LAB / META·SIFT — prompt6 Team Opinions & Implementation Plan

Date: 2026-06-09 · Brief: `.claude/Prompts/prompt6.md` (19 tasks) · Baseline: v2.4.0, 216/216 screening tests passing, repo suite 632 pass / 6 pre-existing `serverStorage.test.js` failures / 7 skips.

Team: Main Claude (manager/integrator), Backend/Auth/DB Developer, Frontend App Developer, Research Engine Developer, QA Developer, Website Manager, Collaboration & Realtime Agent. Each agent independently reviewed the brief against a full codebase reconnaissance (5 mapper agents) before giving the opinions below.

---

## 1. Main Claude — architecture opinion

The 19 tasks are one product idea: **the linked pair (META·LAB `Project` ↔ META·SIFT `ScreenProject`) must behave like one shared workspace** — membership, permissions, link state, notifications, and live updates all flow from it. The single most important architectural decision is what "Review Workspace" physically is:

**Decision: the Review Workspace IS the `ScreenProject` row.** It already carries owner (`ownerId`), members + 17 permission flags (`ScreenProjectMember`), the link (`linkedMetaLabProjectId`), PICO snapshot, status, and audit. `workspaceId == ScreenProject.id`. We will *not* create a physical `ReviewWorkspace` table this round — every agent independently reached the same conclusion (a multi-table data migration mid-release is the largest regression risk in the brief, for zero functional gain). We name the concept in docs and API responses (`relatedWorkspaceId` aliases the ScreenProject id) and defer a physical table.

Everything else follows:

- **Why members "don't see the link" / land on the wrong project**: two one-line ownership bugs — `getMetaLabSummary` filters `ownerId: req.user.id` (members get `linked:false`), and the monolith always selects `projects[0]` on mount, ignoring the `?project=` id that META·SIFT already sends. Fix the source, not the symptom.
- **Why "project not found" on import**: `importRecords` uses the owner-only guard; every member (even leaders) 404s. Convert to access-based guard: outsider → 404 (keep existence-hiding), member without permission → 403. The "stale permission cache" theory is false — access is resolved per-request from the DB.
- **Read-only viewers**: the server already no-ops member autosaves (200 + `skipped` — load-bearing batch contract); the actual bug is the monolith mutates `_readOnly` projects locally. Gate the single client write choke point (`updateProject`), keep the server skip as defense-in-depth, and close the one real server hole (foreign-owner `save()` → 500 instead of 403).
- **Realtime**: SSE with thin "poke" events + polling fallback. No WebSocket (nothing bidirectional to push), no fat payloads (every fat event is a permission decision; pokes make leaks structurally impossible — clients refetch through already-authorized endpoints).
- **Notifications**: persistence-first (DB table + server-authoritative unread count = free logout/login persistence), polling bell cloned from the proven ChatLauncher pattern, SSE poke as an upgrade. Never mounted under rate-limited routers.
- **Monolith strategy**: it has no router and no props; we add a minimal seam (`?project=` query param consumed in the mount effect) instead of a route refactor. New tabs (Overview, Project Control, Methods & Equations) join the existing string-tab system but in a separate sidebar group with `phase:null` so the PRISMA workflow progress math is untouched. Templates removal goes **first** — it deletes the two giant base64 lines that break file tooling.

Sequencing: one additive Prisma migration up-front (Main Claude), then four backend agents + three frontend agents in parallel with strict file ownership, then the realtime agent alone (its emit hooks touch everyone's files), then QA, then reports. The monolith is single-owner, edited in two sequential passes.

## 2. Backend, Auth & Database Developer — opinion

- **Notification model**: new table, indexed `[userId, readAt]` + `[userId, createdAt]`; denormalized actor name/email (actor may be deleted later); store both `relatedMetaLabProjectId` and `relatedScreenProjectId`; no `relatedWorkspaceId` column (workspace = ScreenProject; alias in API). Own router at `/api/notifications` behind `requireAuth` only — **not** under `/api/admin` (60 req/15min) or `/api/auth` (20/15min); bell polling there would self-DoS. All writes scoped `where {id, userId}`; creation best-effort `.catch(()=>{})` — a notification failure must never fail an invite.
- **Login metrics**: dedicated `LoginEvent` model (NOT SecurityEvent rows — no indexes there, and high-volume successes would pollute the forensics table). Write fire-and-forget in the login success path. Rolling windows (24h/7d/30d/90d/365d) per the brief's "past X" wording. Lazy retention sweep (>400d) — no cron infra.
- **lastActive**: write on login + throttled in `requireAuth` via module-level `Map<userId, lastWriteMs>` (only update if >5 min since last write, never awaited). Covers "opens app / saves / screens / sends message" automatically since everything flows through requireAuth. Root cause confirmed: today only profile updates write `lastActive`.
- **Done-today**: new `ScreenProjectStatusEvent` table written only on real status change, in *both* the member-facing and admin status endpoints; metric = `COUNT(DISTINCT projectId) WHERE status='done' AND createdAt >= startOfDay` — distinct-by-project inherently satisfies toggle-twice-counts-once. Don't query ScreenAuditLog JSON for metrics (fragile).
- **Import fingerprint**: SHA-256 of the raw `content` server-side (never trust a client hash), normalize `\r\n→\n` first; nullable additive columns on `ScreenImportBatch`; per-project `findFirst({projectId, fileHash})`; hit → **409** `{error:'duplicate_import', batch:{…}}` unless `force:true`; plus record-level DOI/PMID/normalized-title dedupe. Pushback: never dedupe by filename/size — `pubmed_export.ris` collides legitimately.
- **Viewer read-only**: autosave member path is already correctly enforced server-side (200 + `skipped` no-op). **Do not convert to 403** — the bridge PUTs all projects in one `Promise.all`; one 4xx loses the user's own edits. Close the real holes: client choke point (frontend), `store.save()` foreign-owner 500→403, export honoring read-only flags.
- **403-vs-404**: replace `getOwnedProject` in `importRecords` with `getProjectAccess` + `canImportRecords || isLeader || isOwner`; outsider stays 404. The brief's `canImportStudiesToMetaSift` = existing `canImportRecords` — map, don't duplicate the flag. Audit the other 11 `getOwnedProject` call sites with the same pattern; keep `deleteProject` owner-only.
- **Mod RBAC**: server is already correct (requireAdminOrMod split, DB-verified roles, no hardcoded emails). The 404 is purely `AdminRoute.jsx` checking `role !== 'admin'`. Keep `/metrics` and all `/screening/*` admin-only. Raise/exempt the admin limiter so a mod's console polling doesn't 429 itself.
- **Migration safety**: one additive migration; nullable-or-defaulted columns only (SQLite); no FK cascade from LoginEvent/Notification to User (metrics must survive user deletion — SecurityEvent precedent); add `@@index([linkedMetaLabProjectId])` on ScreenProject (Tasks 3/8/11 all reverse-lookup by it); all new code uses the shared `server/db/client.js` Prisma client.
- **Task 2 atomicity**: auto-create the linked ScreenProject **server-side** inside `projectsController.createProject` (reusing seed + `ensureLeaderMember` internals via a shared helper) — not a second client POST; two client calls can't be atomic and a half-created pair is exactly the broken-link state Task 3 fixes.

## 3. Frontend App Developer — opinion

Not inside UserMenu — invites need an at-a-glance badge, not a buried menu item. Poll clone of `ChatLauncher` (interval, `document.hidden` pause, server-authoritative unread, red 99+ badge). Click = mark read → navigate (`/sift-beta/projects/:pid` SPA; `/app?project=<id>` for META·LAB — coupled to the deep-link fix).
- **Overview/Control tabs**: add to `TABS` with `phase:null`, rendered as a separate "Project" sidebar group above Workflow; filter `TABS.filter(t=>t.phase)` in the workflow map, progress denominator, and "Next step" walker so progress math is untouched. Landing target `setTab("overview")` at all three entry points. Overview content is all in-scope already (created/modified, `_owner/_role`, `stepStatus()`, `readinessCheck()`, studies count, PRISMA numbers, linked summary fetch). Control tab ports the SIFT `ProjectControlTab` pattern (SectionLabel + Card + optimistic toggle with revert); members come from `screeningApi.listMembers(linkedScreenProjectId)` — the workspace is the source of truth; unlinked → "Create & link" card.
- **Methods & Equations**: **HTML/Unicode rendering, emphatically not KaTeX** (~280KB dependency for a fixed, self-authored equation set; Unicode + tiny `<Frac/>` component covers 100% of the inventory). Transcribe from `statistical-validation.md` — do not hand-write new equation prose (that's how the Hedges-g misclaim happened).
- **Read-only gating**: one choke point — early-return in monolith `updateProject` when target is `_readOnly` — kills every write path at once; control hiding/disabling is polish on top. Fix the autosave batch: skip PUT for `_readOnly` projects and switch `Promise.all` → `Promise.allSettled` so one failure can't flip the save indicator for the user's own projects. Persistent "Read-only access" pill in the header.
- **Deep link**: emitter is already correct (`/app?project=${linked.id}`). Fix the receiver: read the param inside the same effect that loads projects (never a separate effect — race), explicit "You don't have access / link broken" panel when the id isn't in the list (never silently select `projects[0]`), `history.replaceState` after consuming. Make the dashboard linked badge clickable; fix RayyanTab's generic `/sift-beta` button to use the linked id.
- **AI hiding**: single `AI_FEATURES_ENABLED=false` flag; `AIButton` returns null; explicitly gate the non-AIButton surfaces (Search AI panel, Extraction modal/toolbar/empty-state, PROSPERO generate, Manuscript draft, AddStudyModal Claude fallback copy, welcome marketing copy). Hide panels entirely — disabled AI buttons invite support questions. Keep `callClaude` infra untouched.
- **Risks**: 5+ tasks touch the 7,352-line monolith — serialize agents or merge conflicts are guaranteed; do Templates removal first (kills the base64 lines that break tooling). Any new tab added without `phase:null` silently corrupts progress UI.

## 4. Research Engine Developer — opinion

- **Methods content as engine-owned data**: `src/research-engine/docs/methods-content.js` exporting `METHODS_CONTENT = [{id, title, equations[], plainEnglish, usedIn, implementedIn, references[], limitations, verified}]`; monolith `MethodsTab` imports it. Structural unit test: every entry has equation/reference/usedIn/limitations; nothing outside the implemented whitelist; `verified:false` renders a "needs verification" badge.
- **The implemented whitelist** (document these and nothing else): IV fixed-effect; Cochran Q + χ² p; I² with bands; DL τ²; DL random-effects; z + two-sided p; 95% CI; **HKSJ (t, k−1)**; **prediction interval (t, k−2, k≥3)**; **Egger's weighted regression**; Duval–Tweedie trim-and-fill (L0); leave-one-out; DFFITS-style influence; subgroup Q-between; MD; SMD (**Cohen's d — see pushback**); log OR / log RR (no continuity correction, null on zero cells); log HR from CI; Fisher z; logit proportion (+0.5 at extremes); log DOR (Haldane +0.5); the 9 conversion recipes (Wan 2014/Hozo 2005); duplicate `scorePair` (0.7 title / 0.15 authors / 0.15 year; DOI/PMID exact ⇒ 100). Closing line: "Not implemented: REML, Paule–Mandel, Peters, Begg, meta-regression, network MA."
- **Canonical references** (real, verified): Cochran 1954; DerSimonian & Laird 1986; Higgins & Thompson 2002; Hartung & Knapp 2001 + Sidik & Jonkman 2002 + IntHout 2014; Higgins/Thompson/Spiegelhalter 2009 + Riley 2011 (PI); Egger 1997; Duval & Tweedie 2000; Viechtbauer & Cheung 2010 (LOO/influence); Borenstein et al. 2009; Cohen 1988; Hedges & Olkin 1985 (pooled-SD standardiser only); Fisher 1921; Glas 2003 + Haldane 1956 (DOR); Tierney 2007 + Parmar 1998 + Cochrane Handbook §6.3/§6.5.2 (HR/conversions); Wan 2014; Hozo 2005; Levenshtein 1966; Jaccard 1912; Abramowitz & Stegun 1964; Numerical Recipes; Lanczos 1964. Label as in-house heuristic / needs-verification: Acklam inverse-normal, scorePair weights & thresholds, logit-SE delta-method, exact Wan variant.
- **Pushback 1 — the SMD is Cohen's d, not Hedges' g.** No small-sample J correction exists in `calculators.js` or the monolith copy, despite code comments and `statistical-validation.md` §9 claiming "Hedges' g". The Methods tab must say Cohen's d; fix the comments/doc lines; do **not** silently add the J correction now (changes every SMD result, invalidates 58 tests; recommend as next step).
- **Pushback 2** — nobody "rounds out" the reference list with plausible extras. The whitelist is the contract; QA diffs the rendered tab against it.
- **Task 2 engine work**: extract `snapshotPico()` helper, use it in auto-create, `linkMetaLab`, and SIFT-side `createProject` (which today produces `picoSnapshot="{}"` forever via the META·LAB "Create & link" path); add lazy snapshot refresh on read (best-effort, compare-before-write).

## 5. Collaboration & Realtime Agent — opinion

- **Transport: SSE + polling fallback, not WebSocket.** All client→server traffic is already REST with cookie auth; realtime here is purely server→client. EventSource is same-origin through the Vite proxy, so the `metalab_session` cookie flows even with `sameSite:'strict'`. Endpoint `GET /api/events` on its own router (never under rate-limited mounts). No compression middleware exists → no flush problem; set `text/event-stream`, `flushHeaders()`, `:hb` heartbeat ~25s, `req.on('close')` cleanup. Test through the :3000 proxy and against `127.0.0.1` (undici ::1 hang).
- **Bus**: in-process `Map<userId, Set<res>>`, one global stream per tab (browser ~6-connection cap — never per-project streams). Single Node process + SQLite — no Redis; document the clustering limitation.
- **"Poke, don't payload"**: events carry only `{type, projectId, at}` — no content. Clients refetch through existing authorized endpoints, each re-checked per request. A fan-out bug can leak *that* something changed, never *what*. Recipients resolved at emit time from active member rows (a just-removed member silently stops receiving). Omit actor identity from decision events entirely (blind-mode safe by construction).
- **Event types**: `project.updated`, `members.changed`, `permissions.changed` (user-targeted), `decision.saved`, `chat.message`, `status.changed`, `handoff.updated`, `notification.created`. Emit hooks live next to existing `writeAudit` calls.
- **Role-change propagation**: server access is per-request from DB (no cache) — propagation is a client problem. On `permissions.changed`, `refreshProject()`; if the refetch 403s, show "access changed" and navigate away. META·LAB side refetches the project list to refresh `_role/_readOnly` annotations.
- **Biggest correctness trap**: META·LAB whole-blob autosave vs realtime refetch — a naive refetch clobbers in-flight local edits, and an applied remote blob followed by local autosave silently destroys the other user's work. Guard: never apply `project.updated` while `hasPendingSave()`; show an "updated by a collaborator — refresh" banner instead.
- **Honest semantics**: "all text updates live" cannot be literally delivered for the PICO/extraction blob without OT/CRDT. Delivered semantics = *fresh-on-clean + conflict banner when dirty*, documented in the realtime architecture doc.
- **Chat**: keep the 4s poll as baseline; `chat.message` poke triggers immediate `listChat(since)` and stretches the poll to ~30s while SSE is healthy. Typing stays in-memory.
- **Invite delivery**: notification rows at addMember + claim-on-register hook for pending (userId-null) invites — without it pending invitees never get their bell; this must be a decided, tested behavior.

## 9. Final chosen implementation plan

**Migration (one, additive)** — `prompt6_notifications_logins_status_fingerprint`: new `Notification`, `LoginEvent`, `ScreenProjectStatusEvent` tables (indexed, no FK-cascade to User); `ScreenImportBatch` + `fileHash/fileSize/importedById/importedByName/parser` (nullable/defaulted); `@@index([linkedMetaLabProjectId])` on ScreenProject.

**API contract (new/changed)**
- `GET /api/notifications` (`?unread=1`, paginated) · `GET /api/notifications/unread-count` · `POST /api/notifications/:id/read` · `/:id/dismiss` · `/mark-all-read`. Notification JSON includes `relatedWorkspaceId` (= ScreenProject id alias), `relatedMetaLabProjectId`, `relatedMetaSiftProjectId`, `app`, `actorName/Email`, `role`.
- `POST /api/projects` accepts `{name, createLinkedSift?:boolean}` → creates Project (+ ScreenProject via shared helper `server/screening/createScreenProject.js` with same owner/title, PICO snapshot, seeded reasons/keywords, owner member row) and returns `{project, linkedScreenProject?}`. Failure to create the SIFT side never rolls back the META·LAB project (warning in response).
- `GET /api/projects` / `GET /api/projects/:id` annotate `_linkedMetaSift: {id, title}` (underscore = stripped on persist) + `_permissions` for shared rows.
- `PATCH /api/projects/:id` allows member rename when workspace perms grant it (owner/leader/canEditMetaLab, not read-only); rename syncs the linked ScreenProject title iff titles were equal before (sync-if-in-sync), and vice versa from SIFT. Documented in api-contract.
- `POST /api/screening/projects` validates `linkedMetaLabProjectId` ownership + snapshots PICO; accepts `alsoCreateMetaLab?:boolean` for the SIFT-side optional flow.
- `GET /api/screening/metalab/:mlpid/summary` membership-aware (members of the linked workspace see `linked:true`).
- `GET /api/screening/projects/:pid` adds `linkedMetaLabProjectTitle`.
- `POST /api/screening/projects/:pid/import`: access-guard (outsider 404; member w/o `canImportRecords`/leader/owner → 403); fingerprint pre-check → **409** `{error:'duplicate_import', batch:{filename, importedAt, importedByName, recordCount}}`; `force:true` overrides; response `{imported, skippedDuplicates, total, batchId}` with DOI/PMID/title record-dedupe always on.
- Same access+perm pattern applied to export (`canExportRecords`), duplicates detect/resolve (`canManageDuplicates`), labels/reasons (leader), listDecisions (member-readable); `deleteProject` stays owner-only.
- `GET /api/admin/metrics` adds `logins:{day,week,month,quarter,year}` (distinct userIds, rolling windows). `GET /api/admin/screening/metrics` adds `doneToday/doneThisWeek/doneThisMonth` (distinct projects). `GET /api/admin/projects` rows add `linkedMetaSift{id,title}` + workspace/owner/status; `GET /api/admin/screening/projects/:id` expanded progress: total/screened/unscreened/included/excluded/maybe/conflicts/duplicates/secondReview/sentToExtraction + per-member progress.
- `GET /api/events` — SSE stream (cookie auth), thin events; clients fall back to polling.
- `addMember` accepts `modules: 'metalab'|'metasift'|'both'` (maps to canView* flags) alongside `preset`; `updateMember` unchanged server-side (already complete) — both apps' UIs call the same endpoints.
